# Conceitos — Etapa 01: BMP280 via I²C

Síntese das discussões de revisão da etapa 01 (diário: `docs/01-bmp280.md`,
código: `firmware/estacao/main/sensor_bmp280.c` e `estacao.c`).

## O sensor: como o BMP280 mede

O BMP280 é um sensor **MEMS piezo-resistivo**. Uma membrana de silício
microscópica fica suspensa sobre uma cavidade de vácuo de referência; a
pressão atmosférica a deforma, e resistores implantados nela mudam de
resistência com a deformação (efeito piezo-resistivo), medidos por uma
ponte de Wheatstone. A temperatura vem de um sensor de silício no próprio
die (tensão de junção varia com a temperatura).

A temperatura tem dupla função: dado ambiente **e insumo da compensação da
pressão** — a resposta da membrana varia com a temperatura, então a fórmula
de pressão consome o `t_fine` derivado da temperatura. Consequência prática
vista no projeto: no módulo defeituoso (ADC de temperatura morto, raw = 0),
a pressão também saía errada (578 vs ~890 hPa).

Caminho do sinal: grandeza física → ponte resistiva → ADC interno →
**contagens brutas (raw)** → fórmula de compensação com coeficientes de
fábrica → °C e Pa.

### Bruto (raw) vs compensado

O raw são as contagens do ADC (20 bits, registradores 0xF7–0xFC) — não
significam nada em °C/Pa por si só. Cada chip sai da fábrica com 12
coeficientes de calibração individuais gravados em NVM (0x88–0xA1), porque
cada membrana MEMS é ligeiramente diferente. O polinômio do datasheet
(§3.11.3) converte raw + coeficientes em valores físicos. Diagnóstico da
etapa: raw de temperatura 0 na fórmula produz ≈ −142 °C — batia com o
sintoma.

## O barramento I²C

### Por que I²C (e não SPI, que o chip também suporta)

- **Economia de pinos**: 2 fios para o barramento inteiro, compartilháveis
  com futuros sensores I²C. SPI usaria 4 fios + 1 CS por dispositivo — e o
  VSPI já está reservado para o LoRa, que precisa da velocidade.
- **Velocidade sobra**: pressão/temperatura mudam devagar; 6 bytes a cada
  segundos cabem folgados nos 100–400 kHz do I²C.
- É a interface padrão de fato de sensores lentos; o módulo GY-BMP280 já
  vem com pull-ups soldados esperando I²C.

### Serial, síncrono, half-duplex

- **Serial**: os bits passam um por vez num único fio de dados (SDA).
- **Síncrono**: há um fio de clock explícito (SCL), gerado pelo master; o
  slave amostra SDA na borda do SCL. Ninguém combina baud rate previamente
  (diferente da UART, serial *assíncrona*). O clock não é "paralelo" no
  sentido de barramento paralelo — é uma linha dedicada de sincronismo.
- **Half-duplex**: existe um único fio de dados; master e slave se revezam
  nele. Full-duplex exigiria dois fios de dados (como MISO/MOSI no SPI).

**SDA** = Serial Data (dados, bidirecional). **SCL** = Serial Clock (clock,
do master).

### O que se envia PARA o sensor

A comunicação não é só leitura — o controle do sensor é feito escrevendo
registradores de configuração:

- `ctrl_meas` (0xF4): oversampling de T e P (1x–16x) e modo
  (sleep/forced/normal);
- `config` (0xF5): filtro IIR e tempo de standby do modo normal;
- `reset` (0xE0): escrever 0xB6 reseta o chip.

Na investigação do módulo defeituoso, `ctrl_meas` foi escrito "na mão" em
várias combinações para testar o chip.

### Open-drain e pull-up

As saídas I²C são **open-drain**: um dispositivo só sabe puxar a linha para
GND ou soltá-la (alta impedância) — ninguém empurra 3,3 V. O **pull-up**
(resistor entre a linha e VCC, ~4,7–10 kΩ) devolve a linha ao nível 1
quando todos soltam. Motivo: o barramento é compartilhado; com push-pull,
dois dispositivos falando ao mesmo tempo fariam curto. Com open-drain, quem
puxa para baixo "vence" (*wired-AND*) — é a base do ACK e da arbitragem.
No projeto há pull-ups em paralelo: os do módulo + os internos da ESP32
(`.flags.enable_internal_pullup = true`), sem problema em barramento curto.

## Endereços: dois conceitos diferentes

1. **Endereço de dispositivo (7 bits, no barramento)**: identifica o chip.
   BMP280: 0x76 ou 0x77. O *scanner* do diário (0x08–0x77) não é varredura
   de memória — é um teste de ACK: transmite cada endereço e vê se alguém
   responde (puxando SDA no 9º clock). Confirma "sensor vivo, fiação certa"
   antes de qualquer driver.
2. **Endereço de registrador (8 bits, dentro do chip)**: a "memória"
   interna do sensor. ID em 0xD0 (0x58 = BMP280), calibração em 0x88–0xA1,
   configuração em 0xF4/0xF5, **resultados em 0xF7–0xFC** (3 bytes de
   pressão + 3 de temperatura, raw). Uma leitura = "escrevo 0xF7" (aponta o
   ponteiro interno) → "leio 6 bytes" (ponteiro auto-incrementa).

### Pinos CSB e SDO (por que ficam em "None")

São pinos de **configuração de hardware** do chip, já resolvidos por
resistores na placa do módulo:

- **CSB**: seleciona o protocolo (alto = I²C; usado como chip-select =
  SPI). O módulo o segura em VCC → nasce em I²C.
- **SDO**: em SPI seria o MISO; em I²C define o último bit do endereço
  (GND = 0x76, VCC = 0x77). O módulo o puxa para GND → 0x76.

Por isso o comentário em `sensor_bmp280.c` cita o SDO mesmo sem fio
conectado: **o pino define o endereço, mas quem o define é o resistor do
módulo**. Como módulos diferentes puxam SDO para lados diferentes, o código
não assume: `detectar_endereco()` faz probe em 0x76 **e** 0x77.

## O código do projeto

### Handle ≠ dado

- **`bus`** (`i2c_master_bus_handle_t`): ponteiro opaco para o estado do
  driver do controlador I²C ("o barramento 0, nos pinos 21/22"). Não
  carrega dado de sensor — é o "descritor de arquivo" do barramento, que
  toda operação (probe/transmit) recebe.
- **`static bmp280_handle_t s_bmp280 = NULL;`**: handle da instância do
  driver do sensor. `static` em escopo de arquivo = privado do `.c`
  (convenção `s_`); `= NULL` = "ainda não inicializado", o que permite o
  guard de `sensor_bmp280_ler()` retornar `ESP_ERR_INVALID_STATE` se for
  chamado antes do init. Após `bmp280_init()`, guarda a estrutura interna
  do driver: dispositivo I²C (barramento + endereço), configuração e os
  **coeficientes de calibração já lidos do chip**.
- `sensor_bmp280_init()` retorna `esp_err_t` e o `return
  bmp280_init(...)` é repasse direto — o driver também retorna `esp_err_t`
  (padrão do IDF: `ESP_OK` = 0 ou código de erro).

### Por que o barramento precisa ser iniciado

`i2c_new_master_bus()` transforma dois GPIOs genéricos num barramento
utilizável — a matriz de GPIO é parte da resposta, não toda:

1. roteia o periférico I²C interno para os pinos 21/22 (matriz de GPIO);
2. configura os pinos (open-drain, pull-up interno);
3. liga/seleciona o clock do periférico;
4. configura o filtro de glitch (`glitch_ignore_cnt = 7` — ruído de
   protoboard);
5. aloca o estado do driver: interrupções e o mutex que serializa o acesso
   entre tasks.

### `#if CONFIG_...` — seleção em tempo de compilação

O `Kconfig.projbuild` declara um `choice` (modo estação / bancada / teste
isolado por sensor). O `menuconfig` grava a escolha no `sdkconfig`; o build
gera `sdkconfig.h` com `#define CONFIG_...` **só do símbolo escolhido**. O
pré-processador C descarta os blocos com condição falsa — o código nem
chega ao compilador. Por isso `estacao.c` tem vários `app_main()`: só um
sobrevive. O sentido é Kconfig → código (o código consome os símbolos via
`sdkconfig.h`), não o contrário.

### Flags de defeito — em tempo de execução (outro mecanismo!)

Não confundir com os `#if`. `pacote.flags` é **por sensor**, um bit cada
(`common/pacote/include/pacote.h`): bit 0 = BMP280, 1 = LDR, 2 = chuva,
3 = DHT11. A cada ciclo, `flags` zera; cada leitura que falha liga seu bit
com OR e zera o campo de dado. Falhas múltiplas **acumulam bits** (BMP280 +
DHT11 → 0x09), e o gateway sabe exatamente quem falhou. O LED, esse sim, é
um resumo geral: `flags == 0` → verde; qualquer bit → amarelo
(`ERRO_SENSOR`); rádio mudo/TX falho → vermelho (`FALHA`).

### Cadência de leitura

Os 2 s do teste isolado são conforto humano, não exigência do BMP280 (no
DHT11, sim: ≥1 s entre leituras). O limite real do BMP280 é o tempo de
conversão: dezenas de ms (datasheet, tab. 14: ~43,2 ms no perfil de máxima
resolução). No modo normal o chip mede em ciclo contínuo (medição +
standby); ler mais rápido que o ciclo só repete o último valor. Poderia ler
a cada ~100 ms — mas pressão muda em escala de horas, daí os 30 s do modo
produção (`CONFIG_ESTACAO_INTERVALO_S`).

### Selftest → Python: o contrato é a linha de log

Não há canal direto. O firmware imprime vereditos em formato **estável**
(`[SELFTEST] bmp280 OK` / `FALHA (código)`); o `tools/bancada.py` abre a
serial, reseta a placa via DTR/RTS (mesmo truque do esptool), aplica a
regex `\[SELFTEST\] (\w+) (OK|FALHA)` por até 25 s e compara com os 8
componentes esperados, saindo com código 0/1 (utilizável em CI). O firmware
não sabe que o Python existe — só honra o formato; mudar a string quebra o
parser.
